# REQUIREMENTS

## 1. 문서 목적

이 문서는 `PROBLEM.md`의 요구사항을 구현 가능한 형태로 구체화하고, 명시되지 않은 정책을 합리적으로 결정하기 위해 작성한다.

본 프로젝트의 우선순위는 다음과 같다.

1. 서버가 빌드되고 실행 가능해야 한다.
2. 핵심 기능(조회, 수강신청, 수강취소, 시간표 조회)이 동작해야 한다.
3. 정원 초과를 방지하는 동시성 제어가 구현되어야 한다.
4. 불명확한 요구사항에 대한 판단 근거가 문서화되어야 한다.

## 2. 명시 요구사항 정리

`PROBLEM.md`에서 직접 확인되는 필수 요구사항은 다음과 같다.

- REST API 서버를 구현한다.
- `GET /health` 엔드포인트는 반드시 `200 OK`를 반환해야 한다.
- 학생 목록 조회 API를 제공한다.
- 교수 목록 조회 API를 제공한다.
- 강좌 목록 조회 API를 제공한다.
- 강좌 목록은 전체 조회와 학과별 조회를 지원한다.
- 강좌 목록 응답에는 정원, 현재 신청 인원, 시간 정보가 포함되어야 한다.
- 수강신청 API를 제공한다.
- 수강취소 API를 제공한다.
- 이번 학기 기준 내 시간표 조회 API를 제공한다.
- 학생은 최대 18학점까지 신청할 수 있다.
- 같은 시간대의 두 강좌를 동시에 신청할 수 없다.
- 정원 1명이 남은 강좌에 100명이 동시에 신청해도 정확히 1명만 성공해야 한다.
- 서버 시작 시 테스트 가능한 초기 데이터가 준비되어야 한다.
- 데이터는 실행 시 동적으로 생성되어야 한다.
- 데이터 규모는 최소 학과 10개, 강좌 500개, 학생 10,000명, 교수 100명을 만족해야 한다.
- API 문서는 `docs/` 디렉터리에 포함되어야 한다.
- 동시성 제어 전략은 `docs/REQUIREMENTS.md`에 설명되어야 한다.

## 3. 구현 범위 결정

이번 구현에서는 평가 우선순위에 맞춰 다음 범위를 1차 목표로 삼는다.

### 포함

- 헬스체크
- 학생 목록 조회
- 교수 목록 조회
- 강좌 목록 조회
- 수강신청
- 수강취소
- 학생별 이번 학기 시간표 조회
- 서버 시작 시 대량 데이터 생성
- 정원 초과 방지를 위한 동시성 제어

### 제외 또는 단순화

- 로그인 및 인증/인가
- 다중 학기 운영
- 선수과목
- 학년별 신청 제한
- 재수강 정책
- 장바구니 기능
- 수강신청 기간/정정 기간의 날짜 정책
- 대기열 기능

제외 사유는 제한된 시간 안에 평가 핵심인 조회, 신청/취소, 정합성, 동시성 제어에 집중하기 위해서다.

## 5. 애매한 요구사항에 대한 결정

### 5.1 인증/인가

구현하지 않는다.

이 과제는 서버 핵심 기능과 동시성 제어가 중심이며, 사용자 인증에 대한 요구사항이 전혀 제시되지 않았다. 따라서 조회 및 신청 API는 테스트 편의를 위해 학생 ID를 직접 전달받는 방식으로 단순화한다.

### 5.2 학기 모델

이번 구현은 단일 활성 학기만 운영한다.

문서와 API에서는 "이번 학기"만을 대상으로 하며, 모든 시간표 조회와 수강신청은 현재 활성 학기에 대해서만 처리한다. 내부적으로는 확장 가능성을 위해 `semester` 필드를 유지한다.

### 5.3 수강신청 대상

학생은 존재하는 강좌에 대해서만 신청 가능하다. 동일 학생의 동일 강좌 중복 신청은 허용하지 않는다.

### 5.4 수강취소 정책

현재 활성 학기에 신청한 내역은 언제든 취소할 수 있다고 가정한다.

취소 마감일 정책은 요구사항에 없고, 핵심 평가 범위와 무관하므로 단순화한다.

### 5.5 재수강 정책

이번 구현에서는 과거 학기 데이터가 없으므로 재수강 개념을 다루지 않는다.

### 5.6 시간표 표현 방식

강의 시간은 비교 가능한 구조와 사람이 읽기 쉬운 문자열을 함께 사용한다.

- 내부 표현: `dayOfWeek`, `startPeriod`, `endPeriod`
- 외부 응답: 예시 `"월 09:00-10:30"`

시간 충돌 검사는 문자열이 아니라 구조화된 시간 슬롯 기준으로 수행한다.

### 5.7 학점 범위

강좌 학점은 1학점, 2학점, 3학점 중 하나로 생성한다. 대부분의 강좌는 3학점으로 생성하되 일부 실험/세미나 과목만 1~2학점으로 배정한다.

이 방식은 현실적인 대학 강의 분포를 단순화한 것이다.

## 6. 비즈니스 규칙 정의

수강신청은 아래 규칙을 모두 만족해야 성공한다.

1. 학생이 존재해야 한다.
2. 강좌가 존재해야 한다.
3. 동일 학생이 동일 강좌를 이미 신청한 상태가 아니어야 한다.
4. 강좌의 현재 신청 인원이 정원보다 작아야 한다.
5. 신청 후 총 신청 학점이 18학점을 초과하지 않아야 한다.
6. 이미 신청한 강좌와 시간표가 충돌하지 않아야 한다.

수강취소는 아래 규칙을 따른다.

1. 학생이 존재해야 한다.
2. 강좌가 존재해야 한다.
3. 해당 학생의 해당 강좌 신청 내역이 존재해야 한다.
4. 취소 성공 시 신청 내역을 제거하고 강좌 현재 신청 인원을 1 감소시킨다.

## 7. 동시성 제어 전략

### 7.1 해결해야 하는 문제

이 과제의 핵심 위험은 단순한 "정원 체크 후 저장" 로직이 동시 요청 상황에서 정원을 초과시킬 수 있다는 점이다.

또한 정원뿐 아니라 다음 문제도 동시에 방어해야 한다.

- 동일 학생이 같은 강좌를 중복 신청하는 경우
- 동일 학생이 서로 다른 강좌를 동시에 신청해 총학점이 18을 넘는 경우
- 동일 학생이 서로 다른 강좌를 동시에 신청해 시간표 충돌이 발생하는 경우
- 신청과 취소가 동시에 발생해 현재 신청 인원이 틀어지는 경우

### 7.2 선택한 전략

애플리케이션 메모리 기반 저장소를 사용할 경우, 다음과 같은 애플리케이션 레벨 락 전략을 사용한다.

- 학생 단위 락
- 강좌 단위 락
- 수강신청/취소 시 필요한 락을 모두 획득한 뒤 검증과 반영을 하나의 임계 구역에서 수행

### 7.3 락 획득 순서

데드락을 피하기 위해 락 획득 순서를 고정한다.

1. 학생 락
2. 강좌 락

여러 강좌를 동시에 다루는 기능이 추가되더라도 ID 오름차순으로 락 순서를 고정한다.

### 7.4 수강신청 처리 방식

수강신청 요청은 아래 순서로 처리한다.

1. 학생 락 획득
2. 강좌 락 획득
3. 중복 신청 여부 검사
4. 정원 검사
5. 총 신청 학점 검사
6. 시간 충돌 검사
7. 신청 내역 저장
8. 강좌 현재 신청 인원 증가
9. 락 해제

이렇게 하면 검증과 반영 사이의 경쟁 상태를 제거할 수 있다.

### 7.5 수강취소 처리 방식

수강취소도 동일하게 학생 락과 강좌 락을 획득한 뒤 다음 순서로 처리한다.

1. 신청 내역 존재 여부 검사
2. 신청 내역 삭제
3. 강좌 현재 신청 인원 감소
4. 락 해제

### 7.6 이 전략의 한계

이 전략은 단일 애플리케이션 인스턴스에서는 유효하지만, 서버를 여러 대로 수평 확장하면 인메모리 락만으로는 정합성을 보장할 수 없다.

멀티 인스턴스 환경에서는 다음 중 하나가 추가로 필요하다.

- 데이터베이스 트랜잭션 + 적절한 격리 수준
- 비관적 락
- Redis 기반 분산 락
- 큐 기반 직렬화

이번 과제에서는 단일 서버 실행을 전제로 인메모리 락 전략을 채택한다.

## 8. 데이터 생성 전략

### 8.1 생성 시점

애플리케이션 시작 시 메모리 저장소를 초기화하면서 데이터를 생성한다. `GET /health`가 `200`을 반환하는 시점에는 조회와 신청 API가 즉시 호출 가능해야 한다.

### 8.2 최소 생성 수량

- 학과 10개 이상
- 교수 100명 이상
- 학생 10,000명 이상
- 강좌 500개 이상

### 8.3 생성 방식

정적 CSV나 SQL 파일을 사용하지 않고, 코드 내 토큰 목록을 조합해 런타임에 생성한다.

예시 전략은 다음과 같다.

- 학과명 후보 목록을 기반으로 학과 생성
- 한국식 성/이름 토큰을 조합해 학생/교수 이름 생성
- 전공 과목명 토큰을 조합해 강좌명 생성
- 학과별 교수와 강좌를 연결
- 시간 슬롯은 요일과 교시 풀에서 생성

### 8.4 현실성 기준

다음과 같은 데이터 생성 원칙을 따른다.

- 학생 이름은 한국식 이름 패턴을 따른다.
- 교수 이름도 현실적인 한국식 이름 패턴을 따른다.
- 강좌명은 실제 대학 강의명에 가까운 형태로 만든다.
- 강좌 코드는 학과 약어와 숫자를 조합해 만든다.

`User1`, `Course1`처럼 의미 없는 데이터는 사용하지 않는다.

### 8.5 정합성 보장

초기 데이터 생성 시에도 비즈니스 규칙이 최대한 유지되도록 한다.

- 강좌 정원은 20~60명 범위에서 생성
- 강좌 학점은 1~3학점 범위에서 생성
- 시간 슬롯은 정의된 범위 안에서 생성
- 초기 신청 데이터가 있다면 정원 초과나 시간 충돌이 없도록 생성

초기 버전에서는 구현 복잡도를 낮추기 위해 신청 내역 없이 시작하고, 모든 강좌의 `enrolled`는 0으로 둔다.

## 9. API 설계 원칙

REST 스타일을 따르며 리소스 중심으로 설계한다.

예상 엔드포인트는 다음과 같다.

- `GET /health`
- `GET /students`
- `GET /professors`
- `GET /courses`
- `POST /enrollments`
- `DELETE /enrollments`
- `GET /students/:studentId/timetable`

강좌 조회는 다음 필터를 지원한다.

- 전체 조회
- `departmentId` 기반 학과별 조회

응답에는 평가자가 테스트 가능한 수준의 식별자와 상태 정보가 포함되어야 한다.

## 10. 오류 응답 원칙

오류 응답은 원인에 따라 명확히 구분한다.

예상 오류 범주는 다음과 같다.

- 잘못된 요청 형식: `400 Bad Request`
- 존재하지 않는 학생/교수/강좌: `404 Not Found`
- 정원 초과, 중복 신청, 학점 초과, 시간 충돌: `409 Conflict`

에러 메시지는 프론트엔드와 평가자가 원인을 이해할 수 있게 구체적으로 반환한다.

예시:

- `COURSE_FULL`
- `ALREADY_ENROLLED`
- `CREDIT_LIMIT_EXCEEDED`
- `SCHEDULE_CONFLICT`
- `ENROLLMENT_NOT_FOUND`

## 11. 테스트 전략

최소한 아래 시나리오는 자동 테스트 대상으로 삼는다.

- 헬스체크 응답
- 학생/교수/강좌 목록 조회
- 정상 수강신청
- 정원 초과 실패
- 동일 강좌 중복 신청 실패
- 18학점 초과 실패
- 시간 충돌 실패
- 정상 수강취소
- 동시 신청 시 정원 1건만 성공하는지 검증

특히 동시성 테스트는 이 과제의 핵심 평가 항목이므로 별도 테스트 코드로 보강한다.

## 12. 향후 확장 고려사항

이번 구현 범위에서는 제외하지만, 추후 다음 방향으로 확장 가능하도록 설계한다.

- 인증/인가 추가
- 다중 학기 지원
- 선수과목과 재수강 정책 추가
- 분산 환경에서의 동시성 제어
- 영속성 데이터베이스 도입
- OpenAPI 기반 Swagger 문서 자동화

## 13. 결론

이 프로젝트는 "수강신청 기능 제공" 자체보다 "경쟁 상태에서도 정원이 절대 초과되지 않도록 보장하는 것"이 더 중요한 과제다.

따라서 구현은 다음 원칙을 따른다.

- 단일 서버 기준에서 확실한 정합성을 우선한다.
